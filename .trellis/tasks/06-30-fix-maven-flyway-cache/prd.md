# Fix Maven Flyway dependency cache

## Goal

Resolve the local Java/Maven build path error caused by a failed cached download of `org.flywaydb:flyway-core:jar:8.0.5`, so `km-backend` can resolve dependencies and compile diagnostics can be triaged from a clean dependency state.

## Background

- Java language server previously reported `Missing artifact org.flywaydb:flyway-core:jar:8.0.5` with a cached transfer failure caused by `磁盘空间不足`.
- `km-backend/pom.xml:86` declares `org.flywaydb:flyway-core`; the version is managed by the Spring Boot parent in `pom.xml:14`.
- New diagnostics in `km-backend/src/main/java/com/km/client/KmAiClient.java:28` report missing `getData()`, `getCode()`, `getMessage()`, and `log` symbols.
- Those symbols are generated by Lombok: `km-backend/src/main/java/com/km/dto/ai/AiApiResponse.java:5` uses `@Data`, `km-backend/src/main/java/com/km/client/KmAiClient.java:13` uses `@Slf4j`, and `km-backend/pom.xml:90` declares Lombok.
- The repository uses Lombok broadly across backend source files, so the `KmAiClient` errors are more likely an annotation-processing/classpath symptom than an isolated source defect until Maven compile proves otherwise.

## Requirements

- Treat this as a local environment/dependency-cache repair, not an application code change.
- Remove stale or failed Maven cache markers for `org.flywaydb:flyway-core:8.0.5` without altering unrelated repository files.
- Force Maven to retry dependency resolution for `km-backend` after cleanup.
- Run Maven from a clean dependency state before deciding whether the `KmAiClient` Lombok diagnostics require source changes.
- Preserve any unrelated dirty worktree changes.
- Do not change Java source or project configuration unless Maven compile still reproduces a source-level failure and the task scope is explicitly expanded.

## Acceptance Criteria

- [ ] The missing `flyway-core-8.0.5.jar` cache failure is cleared from the local Maven repository.
- [ ] Maven can resolve `km-backend` dependencies, or any remaining failure is reported with its current root cause.
- [ ] `mvn -pl km-backend -am compile -U` succeeds, or the first remaining compile failure is reported with evidence.
- [ ] `KmAiClient` Lombok-related diagnostics are classified as resolved, IDE-only annotation-processing issues, or build-reproducible source errors.
- [ ] No source code or project configuration is changed unless necessary to fix the dependency resolution issue.

## Out of Scope

- Replacing Lombok-generated accessors or loggers with hand-written code as a workaround.
- Changing Flyway or Spring Boot dependency versions without evidence that the declared version is invalid.
- Broad backend refactors unrelated to dependency resolution and compile triage.
