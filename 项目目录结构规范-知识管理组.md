## 一、与大组仓库的关系

大组采用 **单仓多组** 协作（见 `README.md`）：

```
develop
└── feat-a          ← 知识管理组集成分支（组长维护）
    ├── feat/a-login
    ├── feat/a-knowledge-base
    └── ...
```

| 项 | 知识管理组约定 |
|----|----------------|
| 组代码 | **a** |
| 集成分支 | `feat-a` |
| 功能分支 | `feat/a-<功能>`，例 `feat/a-doc-upload` |
| Commit scope | `feat(a): ...` / `fix(a): ...` |
| 前端 | 模块页走 `feat/a-*`；若有大组统一 `frontend` 分支，通用布局走 frontend |

**本规范描述 `km-platform` 仓库内部结构**；合入 `feat-a` 时保持目录不变，仅调整根路径（若大组要求放在 `modules/km/` 下，由总 leader 统一规定）。

## 四、后端标准分层规范

采用与问答组一致的 **12 层思想**，数据访问层用 **MyBatis**（非 JPA）。

```
km-backend/src/main/java/com/km/
├── KmApplication.java           # 1. 启动类
├── config/                      # 2. 配置类
├── controller/                  # 3. REST（按领域分子包）
├── service/                     # 4. 业务接口
│   └── impl/                    #    实现类
├── repository/                  # 5. MyBatis Mapper 接口（@Mapper）
├── entity/                      # 6. 表映射实体
├── dto/
│   ├── request/                 # 7. 请求体
│   └── response/                # 8. 内部/Feign 响应
├── vo/                          # 9. 返回前端的视图对象
├── mapper/                      # 10. MapStruct 转换器
├── exception/                   # 11. 全局异常 + ErrorCode
├── security/                    #    JWT（本组独有，放 security 包）
├── pipeline/                    #    文档处理（本组独有）
└── client/                      # 12. 外部 HTTP 客户端（嵌入/重排 API）
```

### 4.1 各层说明

| 层 | 作用 | 注意事项 |
|----|------|----------|
| **controller** | 对外 REST | 路径前缀 `/api/v1/`；返回 `ApiResponse<VO>` |
| **service** | 业务逻辑 | 不直接写 SQL；编排 pipeline、repository、client |
| **repository** | MyBatis Mapper | 接口在 `repository/`，XML 在 `resources/mapper/` |
| **entity** | 表映射 | 不带 JPA 注解时用普通 POJO + MyBatis 映射 |
| **pipeline** | 异步文档处理 | 仅 document 相关；consumer 里调 parser → chunker → embedder |
| **client** | 调 SiliconFlow 等 | 不要散落在 service 里写裸 HttpClient |

### 4.2 数据流转

```
前端 → Controller(Request DTO) → Service → Repository / Pipeline / Client
                                              ↓
                                    MapStruct → VO → ApiResponse
```

### 4.3 API 路径约定

| 领域 | 路径示例 |
|------|----------|
| 认证 | `POST /api/v1/auth/login` |
| 知识库 | `/api/v1/knowledge-bases` |
| 文档 | `/api/v1/knowledge-bases/{kbId}/documents` |
| 检索 | `POST /api/v1/search` |
| 配置 | `/api/v1/admin/config/embedding` |
| 统计 | `GET /api/v1/stats/summary` |

## 六、DTO / VO / Entity 规范

与问答组一致，四层分离：

| 层 | 包 | 示例 |
|----|-----|------|
| Entity | `entity/` | `KnowledgeBase`, `Document`, `Chunk` |
| Request | `dto/request/` | `CreateKnowledgeBaseRequest`, `SearchRequest` |
| Response | `dto/response/` | `SearchResponse`, `DocumentProcessResponse` |
| VO | `vo/` | `KnowledgeBaseVO`, `SearchResultVO`, `ChunkVO` |

```java
// MapStruct 示例
@Mapper(componentModel = "spring")
public interface KnowledgeBaseMapper {
    KnowledgeBaseVO toVO(KnowledgeBase entity);
    KnowledgeBase toEntity(CreateKnowledgeBaseRequest request);
}
```

**禁止：** Controller 直接返回 Entity；VO 中不暴露 `passwordHash`、完整 `apiKey`。

## 十、数据库规范

### 10.1 库表（MySQL，单库 `km_db`）

| 表名 | 说明 |
|------|------|
| `user` | 用户 |
| `knowledge_base` | 知识库 |
| `document` | 文档元数据 + 状态 |
| `chunk` | 切片（content, chapter_path, vector_id） |
| `system_config` | KV 配置（嵌入/重排/解析器） |

向量数据在 **Milvus/ES**，不在 MySQL 存全文向量。

### 10.2 字段约定

| 类型 | 规则 | 示例 |
|------|------|------|
| 表名 | 小写单数或复数（组内统一） | `knowledge_base`, `document` |
| 列名 | snake_case | `created_at`, `kb_id` |
| 主键 | `id` VARCHAR(36) UUID | — |
| 时间 | `created_at`, `updated_at` | DATETIME |
| 软删（可选） | `deleted` TINYINT | MVP 可物理删 |

### 10.3 Flyway

- 路径：`km-backend/src/main/resources/db/migration/`
- 命名：`V1__init_km.sql`, `V2__add_document_tags.sql`
- **禁止**手改已执行过的 migration
