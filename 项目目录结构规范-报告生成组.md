## 一、 与大组仓库的关系

本小组在大组中代码代表为 **c**，采用 **单仓多模块** 或 **独立仓双向同步** 协作：

```
develop
└── feat-c          ← 报告生成组集成分支（组长谢武林维护）
    ├── feat/c-report
    ├── feat/c-admin
    └── ...
```

| 维度 | 报告生成组约定 |
|----|----------------|
| 组代码 | **c** |
| 集成分支 | `feat-c` |
| 功能分支 | `feat/c-<功能>`，例 `feat/c-docx-export` |
| Commit scope | `feat(c): ...` / `fix(c): ...` |
| 前端部分 | 模块页面走本组 `report-frontend`；通用组件/布局可合并至大组统一的 `frontend` 分支 |

## 三、 模块职责与默认端口

| 模块/服务 | 默认端口 | 职责 | 依赖介质 |
|------|----------|------|------|
| **report-backend** | **8092** | 认证、大纲生成、SSE内容推送、Word规范化导出、统计 | MySQL / Redis |
| **report-frontend** | 5173 | 登录注册、大纲编辑、工作台渲染、管理员设置 | — |
| **MySQL (库名: report_db)**| 3306 | 存储用户信息、历史生成记录（JSON大纲与内容）、配置 | — |
| **Redis** | 6379 | 用户 Session 缓存、大纲流式生成状态 | — |

## 四、 后端标准分层规范

保持與大组一致的 **12层结构** 思想，由于 1 周时间极其宝贵，不使用繁琐的 JPA，统一采用 **MyBatis** 配合轻量 SQL 运作。

```
report-backend/src/main/java/com/report/
├── ReportApplication.java       # 1. 启动类
├── config/                      # 2. @Configuration 配置（安全、跨域、大模型）
├── controller/                  # 3. 对外 REST 控制层
├── service/                     # 4. 业务逻辑层接口
│   └── impl/                    #    业务接口实现
├── repository/                  # 5. MyBatis Mapper 接口 (@Mapper)
├── entity/                      # 6. 表映射实体
├── dto/
│   ├── request/                 # 7. 入参校验对象 (CreateReportRequest)
│   └── response/                # 8. 内部传输对象
├── vo/                          # 9. 视图渲染对象 (ReportRecordVO)
├── mapper/                      # 10. MapStruct 对象互转
├── exception/                   # 11. 全局异常处理
└── engine/                      # 12. 报告生成引擎（本组特有）
```

## 五、 报告生成与导出引擎规范（报告组特有核心包）

在 `report-backend` 的 `engine/` 包下，集中处理 AI 交互与文档排版逻辑，避免业务污染：

```
engine/
├── outline/
│   ├── OutlinePromptBuilder.java    # 大纲 Prompt 生成器
│   └── OutlineParser.java           # 解析 AI 返回的 JSON 大纲并重新编号
├── content/
│   ├── ContentPromptBuilder.java    # 各章节流式正文 Prompt
│   └── SseStreamEmitter.java        # 封装前端 SSE 连接，向前端逐字 push 进度和内容
└── exporter/
    ├── DocxExporter.java            # Word 导出主入口
    ├── StyleConfigurator.java       # 统一样式控制（仿宋、黑体、段落、页眉页脚）
    └── NumberingManager.java        # 处理图表编号（全局编号 vs 按章编号）
```

## 六、 数据库设计规范

由于开发周期短，数据表设计极致精简，单库 `report_db`。

### 10.1 核心表结构

| 表名 | 对应实体 | 说明 |
|------|------|------|
| `user` | `User` | 用户注册登录 |
| `report_record` | `ReportRecord` | 报告生成记录（其中大纲和正文内容分别以 **JSON 格式**存入数据库长文本字段，免去联表开发） |
| `system_config` | `SystemConfig` | 存放 LLM API 地址、API Key、选用的模型等配置 |

### 10.2 关键字段约束

*   表名、列名使用小写下划线（snake_case）命名，如 `created_at`。
*   主键使用 `id` (VARCHAR(36))，采用 UUID 避免主键冲突。
*   时间字段统一使用 `created_at`、`updated_at` (DATETIME)。

## 七、 1周极速自查清单

在提交 MR (Merge Request) 合入本组集成分支 `feat-c` 前，组员请务必对照自查：

- [ ] 包结构命名绝对规范，没有在根目录下乱建其他包。
- [ ] 所有对大模型的请求和 DOCX 的样式配置，全部统一写在 `engine/` 下，严禁散落在业务 Controller 中。
- [ ] 导出的 DOCX 样式必须包含：仿宋正文、黑体标题、标准的图表序号。
- [ ] SSE 推送的事件名称和前端接收完全对齐（`progress` / `content` / `done`）。
- [ ] 不提交 `.env`（仅提交 `.env.example` 模板）、不提交 `target/` 和 `node_modules/`。
- [ ] 每次 Commit 信息完全遵循大组规范，如：`feat(c): 增加报告导出Word及排版样式配置`。
